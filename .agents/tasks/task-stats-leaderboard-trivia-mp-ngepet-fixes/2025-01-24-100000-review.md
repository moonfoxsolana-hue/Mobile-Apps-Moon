# Stats, Trivia Multiplayer Rooms, and Ngepet Fixes

This change adds statistics/leaderboard overlays to all three quiz-style games (Intuition, Logical, Trivia), introduces a full multiplayer room system for Trivia with lobby, real-time polling, in-room gameplay with countdown timer, and leaderboard results, and fixes five behavioral issues in the Ngepet game (mystery box re-selection, auto-redirect on game end, state reset on new match, candle animation, image centering). The implementation spans 12 files with 2208 lines added across models, API service, repository, ViewModels, and Composable screens.

**Watch for:** The room countdown timer is purely cosmetic ‚Äî it reaches zero without auto-submitting or penalizing, so players can wait indefinitely (confirmed). The `startResultPolling` loop re-POSTs `finishTriviaRoom` every 3 seconds as its poll mechanism, which is a write endpoint being used for polling reads (confirmed). The `exitRoom()` function silently swallows the API call failure, meaning the server may not know a player left if the call fails (confirmed).

## High-level view

The statistics and leaderboard feature follows a consistent pattern across all three ViewModels: add `showStats`/`showLeaderboard` toggles, lazy-load on first open, and render as full-screen overlays in each game screen. The API surface is narrow (one GET endpoint per game for statistics, one for leaderboard).

The Trivia room multiplayer is the largest piece. It introduces a state machine with four phases (ROOM_LIST, ROOM_LOBBY, ROOM_PLAYING, ROOM_RESULT) managed entirely in TriviaViewModel via `roomPhase`. Transitions between phases are driven by polling `getTriviaActiveRoom` every 3 seconds during the lobby, and by direct API responses during gameplay. The room model layer is 18 data classes supporting the full lifecycle from creation through result display.

The Ngepet fixes address five distinct UX problems. The mystery box re-selection fix preserves `selectedHiddenItemId` across wrong guesses so subsequent taps go straight to the item guess dialog. The auto-redirect to lobby triggers via `clearGuessResult()` when `isEnd == true` and via `refreshMatchDetail()` detecting closed/end status from the server ‚Äî a dual-path approach providing redundancy. The state reset ensures `guessedItemNames` clears when joining or reconnecting to a match.

## Trivia room state machine and polling architecture

The room lifecycle is managed as a single `roomPhase` enum on `TriviaUiState`. Lobby polling calls `getTriviaActiveRoom` every 3 seconds and inspects `response.state`. When it detects "playing", it stops the poll, copies the first question into state, and returns from the coroutine. This is the mechanism by which non-host players learn the game has started.

The result polling is architecturally questionable. It calls `finishTriviaRoom` ‚Äî a POST endpoint ‚Äî every 3 seconds to check whether `roomFinished == true`. This means the client is repeatedly telling the server "I'm done" as a way to ask "is everyone done?". If the server has any side effects per-call (logging, counter increments, rate limiting), this will cause issues at scale. If the server is idempotent, it works but is semantically wrong ‚Äî a GET or the existing `getTriviaActiveRoom` endpoint would be appropriate here.

## Room countdown timer without enforcement

The room playing phase includes a countdown bar that ticks from 1.0 to 0.0 over 15 seconds:

```kotlin
val totalDuration = 15_000L
val stepDelay = 50L
val steps = totalDuration / stepDelay
val decrement = 1f / steps
repeat(steps.toInt()) {
    delay(stepDelay)
    countdownProgress = (countdownProgress - decrement).coerceAtLeast(0f)
}
```

When `countdownProgress` reaches zero, nothing happens. The player can still select and submit at any point. The single-player Trivia mode has the same visual-only countdown, so this may be intentional consistency, but in a multiplayer context where other players are waiting, an unforced timer means one player can stall the room indefinitely. At minimum, auto-submitting the currently-selected answer (or a blank/skip) at zero would prevent stalling.

## exitRoom and error swallowing

```kotlin
fun exitRoom() {
    val roomId = _uiState.value.currentRoom?.id ?: return
    viewModelScope.launch {
        try {
            val request = TriviaRoomExitRequest(roomId)
            gamesRepository.exitTriviaRoom(request)
        } catch (_: Exception) { }
        stopPolling()
        _uiState.value = _uiState.value.copy(...)
        loadRoomList()
    }
}
```

The exit call fires and the result is discarded regardless of success or failure. Local state is unconditionally cleared. If the network call fails, the server still considers this player in the room. For a room with `maxPlayers` limits, this means a ghost player occupies a slot until the server times them out (if it does). The same fire-and-forget pattern appears in `readyPlayer()` and `kickPlayer()`, though those at least surface exceptions to the error state rather than silencing them.
