# Intruder hidden-guess two-step flow with sound effects and UI polish

The intruder's guess action is changed from a single item-pick dialog to a two-step flow: choose a mystery box (hidden item), then choose which item to guess inside it. The `hidden_item_id` field the backend already required is now actually sent. Alongside this, a `SoundManager` streams MP3 files from the web server on guess outcomes, a shake animation conveys wrong-guess feedback, and the notification system moves from a static bottom `Text` to an animated top overlay with auto-dismiss.

Watch for: SoundManager leaks a `MediaPlayer` instance when `onCompletion` fires before assignment completes (confirmed). `playUrl` racing itself can orphan a prepared-but-unreleased player (likely). Sound playback depends on hard-coded external URLs with no offline or failure-surface handling for the user (confirmed).

## High-level view

The intruder flow now gates on hidden-item selection before item guessing. State flags in the ViewModel (`showHiddenItemSelection`, `showGuessItemDialog`, `selectedHiddenItemId`) control which dialog renders. The concern is that `selectedHiddenItemId` being nullable means a null-safe `?.let` in the UI silently drops the guess if state gets out of sync.

`SoundManager` wraps `MediaPlayer` for URL-based streaming. It creates and releases players one at a time, but the `onCompletionListener` calls `release()` on the player instance while the class also holds a reference to it, creating a dangling-reference window. Since a new `SoundManager` is `remember`ed per `GuessResultDialog` appearance, rapid dismiss-and-reshow cycles can stack unreleased players.

The notification overlay replaces a plain `Text` at `BottomCenter` with an `AnimatedVisibility` card at `TopCenter`. Both message and error are cleared by a single `LaunchedEffect` with a 3-second delay, keyed on both values. If a message appears and an error follows within 3 seconds, the existing coroutine won't restart (same key instance) and the error may not get its full display window.

1. **MediaPlayer dangling reference** ‚Äî `onCompletionListener` releases the player but `SoundManager.mediaPlayer` still points to it. A subsequent `release()` call (from `DisposableEffect`) double-releases. Null out the field in the completion callback or skip release in `onCompletion` and let the class manage lifecycle. (confirmed)
2. **Rapid playUrl self-race** ‚Äî If `playUrl` is called while `prepareAsync` is in flight on the current player, the old player is released before it fires `onPrepared`, which is fine, but the new player assignment can collide with the old `onCompletion` callback nulling/releasing. Guard with a synchronized block or use `ExoPlayer` which handles this natively. (likely)
3. **Hard-coded sound URLs with no fallback** ‚Äî Network failure silently swallows the error via `onErrorListener { true }`. The user gets no audible or visual feedback that sound failed. Consider a local fallback or at minimum log at warn level instead of swallowing. (confirmed)
4. **LaunchedEffect key collision for message/error** ‚Äî `LaunchedEffect(uiState.message, uiState.error)` won't re-launch if only one of the two values changes while the coroutine is already running. A rapid message-then-error sequence won't reset the 3-second timer. Use a monotonically increasing counter or separate effects. (likely)
5. **Nullable selectedHiddenItemId silent drop** ‚Äî In `ItemSelectionDialogWithGreyed`, the guess is fired inside `uiState.selectedHiddenItemId?.let`. If the state is null (e.g., due to a race with `shouldResetSelection`), the user's tap does nothing with no feedback. Add an error toast or prevent the dialog from showing when ID is null. (possible)
6. **Shake won't re-trigger on identical responses** ‚Äî `LaunchedEffect(guessResult)` won't re-fire if two consecutive wrong guesses produce structurally identical response objects. Key on a counter or include a timestamp/nonce in the response. (likely)

## SoundManager lifecycle and threading

`SoundManager` stores a single `mediaPlayer` reference. On each `playUrl` call it releases the previous instance and creates a new one. The `onCompletionListener` also calls `it.release()` on that same instance. If the sound plays to completion naturally, the player is released by the callback ‚Äî but `SoundManager.mediaPlayer` still holds the (now-released) reference. When `DisposableEffect` fires `soundManager.release()`, it calls `release()` again on the dead player. `MediaPlayer.release()` on an already-released instance is technically safe on most AOSP builds, but some OEM forks throw `IllegalStateException`.

If the user dismisses and re-opens the dialog rapidly, two `SoundManager` instances coexist (the first's `DisposableEffect` hasn't fired yet). Each holds its own `MediaPlayer`. This produces overlapping audio and delays resource cleanup until Compose disposal catches up.

## Hidden-item selection flow ‚Äî empty state gap

The `HiddenItemGridDialog` filters to `openItems = hiddenItems.filter { it.status == "open" }`. If all boxes are used, the grid renders empty ‚Äî the user sees only the "Batal" button with no explanation. A "no boxes remaining" message or disabling the trigger button when `openItems` would be empty prevents dead-end UI.

## Shake animation re-trigger failure

The shake is triggered by `LaunchedEffect(guessResult)`. Compose compares the key by structural equality. If two consecutive wrong guesses produce the same `NgepetGuessResponse` shape (`isCorrect=false`, `isEnd=false`, same null fields), the effect doesn't re-launch and the second wrong guess gets no animation. Key on `uiState.intruderGuessCount` instead, which increments on every guess.

## Floating notification timing edge case

The `LaunchedEffect` keyed on `(uiState.message, uiState.error)` launches a single coroutine that clears both after 3 seconds. If a second message arrives with a different string value, Compose cancels the old coroutine and starts fresh ‚Äî so varied messages behave correctly. The failure case is identical consecutive messages (same string value twice): the key doesn't change, the effect doesn't re-launch, and the second notification is invisible. Use a monotonic counter as an additional key to guarantee re-launch.
