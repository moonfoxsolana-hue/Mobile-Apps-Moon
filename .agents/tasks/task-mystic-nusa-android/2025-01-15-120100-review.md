# Mystic Nusa Android app — critical fixes applied (v2)

The six critical issues from the v1 review have been addressed. The wallet address is now submitted as the `AirdropClaimRequest` body during claim. The OkHttp auth interceptor reads from an in-memory `cachedToken` field instead of calling `runBlocking` on DataStore. Logging is gated behind `BuildConfig.DEBUG`. A 401 response interceptor clears the cached token. Error body parsing extracts the server's `message` from JSON in all repositories. The claim button disables itself via `hasClaimed` state after a successful first claim.

**Watch for:** The 401 interceptor clears only the in-memory token cache without clearing DataStore or signaling the UI — after a restart the expired token is reloaded and used again until the next 401 (confirmed). A race between `MysticNusaApp.loadCachedToken()` and the first API call could result in one unauthenticated request on cold start (possible).

**Verdict**: APPROVED

## High-level view

The auth interceptor now reads a `@Volatile var cachedToken` directly, populated at app startup via `applicationScope.launch { tokenManager.loadCachedToken() }`. This introduces a narrow startup race: if the splash screen's 1500ms delay completes before the IO coroutine finishes reading DataStore, the first request could fire with a null token. In practice, DataStore reads finish in 5-50ms, so the splash delay acts as a de facto synchronization barrier.

The 401 interceptor nulls the in-memory token so subsequent requests drop the bearer header. Because `clearCachedToken()` does not clear the DataStore, a process-death restart reloads the stale token and hits 401 once more before the interceptor clears it again — a benign one-request-per-restart cost, not a blocking issue.

<summary>Issues (2)</summary>

1. **401 clears memory but not DataStore** — After a 401, `clearCachedToken()` nulls the in-memory field but leaves the expired token in DataStore. On next app launch, `loadCachedToken()` restores it and one request will fail before the interceptor clears it again. Non-blocking for ship, but calling `clearAll()` in the interceptor via a fire-and-forget coroutine would eliminate the restart loop. (confirmed)
2. **Startup race between token load and first request** — `loadCachedToken()` runs on `Dispatchers.IO` in `MysticNusaApp.onCreate()`. If any API call fires before that coroutine completes, the request goes out without a token. The 1500ms splash delay makes this practically impossible, but the invariant is implicit rather than guaranteed. (possible)

## 401 interceptor: memory-only clear

```kotlin
private val unauthorizedInterceptor = Interceptor { chain ->
    val response = chain.proceed(chain.request())
    if (response.code == 401) {
        tokenManager?.clearCachedToken()
    }
    response
}
```

`clearCachedToken()` sets `cachedToken = null` and nothing else. The DataStore file still contains the old token. On a cold start, `loadCachedToken()` reads it back into memory and the first request after restart will carry the stale bearer, trigger another 401, and clear it again. The user may briefly see a "Failed to load profile" error that resolves on retry.

The interceptor cannot call a suspending function directly, but a fire-and-forget `CoroutineScope(Dispatchers.IO).launch { clearAll() }` inside `clearCachedToken()` would close the gap. Acceptable to defer.

## Startup token load timing

```kotlin
// MysticNusaApp.kt
override fun onCreate() {
    super.onCreate()
    val tokenManager = TokenManager(this)
    RetrofitInstance.init(tokenManager)
    applicationScope.launch {
        tokenManager.loadCachedToken()
    }
}
```

The `applicationScope.launch` dispatches on `Dispatchers.IO`. Meanwhile the Activity starts, the Compose tree inflates, and SplashScreen adds a 1500ms delay before navigating. If the splash delay were removed or reduced in future, requests could fire with a null token.
