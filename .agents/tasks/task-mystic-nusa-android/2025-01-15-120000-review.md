# Mystic Nusa Android app — full greenfield build

A complete native Android app (Kotlin, Jetpack Compose, Material3) was built from scratch implementing the Mystic Nusa Web3/crypto community platform. The app covers authentication, a content-rich home page, user profile with token history, airdrop claiming, news/stories feeds, a staking system, and six games (Trivia, Logical/IQ, Intuition, Tarot Ritual, Ular Tangga, Ngepet Online). Architecture is MVVM with Retrofit for networking, DataStore for token persistence, and Navigation Compose for routing.

**Watch for:** `runBlocking` inside an OkHttp interceptor will block the network thread on every request (confirmed); auth token is stored in cleartext DataStore with no expiry or refresh mechanism (confirmed); the hardcoded production base URL ships with BODY-level HTTP logging enabled (confirmed); there are zero unit or instrumentation tests (confirmed); wallet address submitted during airdrop claim is validated client-side but never sent to the backend (likely).

## High-level view

The networking layer uses a singleton Retrofit instance whose auth interceptor calls `runBlocking` to read the token from DataStore on every HTTP call. This blocks the OkHttp dispatcher thread and can cause ANRs under slow I/O. The interceptor also runs for unauthenticated routes (login, register), attaching a null bearer header is harmless but the `runBlocking` cost still applies.

Token persistence stores the bearer token in a plain-text Preferences DataStore file. There is no token expiry check, no refresh token flow, and no encrypted storage. If the backend token expires the user will see cryptic network errors until they manually log out and back in.

The `RetrofitInstance` singleton initializes `OkHttpClient` eagerly at class-load time — including the logging interceptor at `Level.BODY` — which will dump every request/response body (including tokens) to logcat in production builds. No build-type gating exists.

The claim screen validates the Solana wallet address client-side with a regex but never submits it to the API — `claimFirst()` calls `POST airdrop/claim` with no body. The validation gives the user confidence their address is recorded, but the address is discarded.

Repository and ViewModel layers follow a consistent pattern but create fresh instances in every composable via `viewModel(factory = ...)` with `remember { TokenManager(...) }`. This means each screen that needs auth creates its own TokenManager instance (though they share the same underlying DataStore file, so correctness is preserved, just wasteful in object allocation and subscription overhead).

The six games are all implemented through a single `GamesRepository` — ~393 lines of mechanically identical try/catch wrappers. The monolithic repository makes it impossible to mock or test individual games in isolation.

<summary>Issues (8)</summary>

1. **`runBlocking` in OkHttp interceptor** — Replace with a suspending interceptor (OkHttp 4.x `Interceptor` with a `runBlocking` is an anti-pattern). Use a `TokenProvider` interface that caches the token in-memory and updates it asynchronously, or switch to an `Authenticator` that only fires on 401.
2. **BODY-level logging in production** — Gate `HttpLoggingInterceptor.Level.BODY` behind `BuildConfig.DEBUG`. Currently leaks tokens and user data to logcat on release builds.
3. **No token refresh or expiry handling** — Add a 401-response interceptor or Authenticator that clears the token and navigates to login, or implement a refresh-token flow. Currently expired tokens produce silent failures.
4. **Cleartext token storage** — Migrate from Preferences DataStore to EncryptedSharedPreferences (Jetpack Security) or at minimum encrypt the token value before storing it.
5. **Wallet address not submitted during airdrop claim** — The `ClaimScreen` validates a wallet address and calls `claimFirst()`, but `AirdropRepository.claimFirst()` calls `POST airdrop/claim` with no request body. Either pass the wallet address in the request or call `updateWallet()` before claiming.
6. **Zero test coverage** — No unit tests, no instrumentation tests. At minimum, ViewModel logic (state transitions, error handling) and repository response parsing should be covered.
7. **Monolithic GamesRepository** — Split into per-game repositories (TriviaRepository, NgepetRepository, etc.) to enable isolated testing and reduce cognitive load.
8. **Multiple TokenManager instances per composition** — Hoist a single `TokenManager` from `MysticNusaApp` (or use a DI framework) instead of creating one per screen via `remember { TokenManager(context) }`.

## `runBlocking` on the OkHttp dispatcher thread

In `RetrofitInstance.kt`, the auth interceptor reads the token via:

```kotlin
private val authInterceptor = Interceptor { chain ->
    val token = runBlocking {
        tokenManager?.getToken()
    }
    ...
}
```

`getToken()` calls `context.dataStore.data.first()`, which suspends while reading from disk. Wrapping this in `runBlocking` on OkHttp's dispatcher thread means every network call (including concurrent ones) blocks a limited pool thread. Under poor disk I/O (cold start, low memory) this can cascade into ANRs. The fix is to cache the token in a `var` that's updated by a coroutine collecting `tokenFlow`, so the interceptor reads a volatile in-memory value with zero blocking.

## Auth token lifecycle gaps

The `SplashScreen` checks whether a token exists and routes to Home or Login accordingly. But there is no validation that the token is still accepted by the backend. If the server has revoked or expired the token:

- Every API call will fail with a 401 (or whatever the backend returns).
- The repositories map this to `Result.failure(Exception("... failed: 401"))`.
- The user sees "Failed to get profile: 401" with no automatic redirect to login.

The `logout()` function in `AuthViewModel` calls `authRepository.logout()` (which hits `POST auth/logout`) and then clears local storage. But if the token is already invalid, that logout call will itself fail — and the coroutine swallows the failure silently, which is acceptable since `clearAll()` still runs. The more critical gap is the absence of an automatic session-expiry handler.

## Hardcoded production URL with debug logging

```kotlin
private const val BASE_URL = "https://mysticnusa.com/api/"

private val loggingInterceptor = HttpLoggingInterceptor().apply {
    level = HttpLoggingInterceptor.Level.BODY
}
```

`Level.BODY` logs the full request and response bodies. For auth responses this includes the `access_token`. For profile responses this includes email, wallet address, and token balances. This ships in all build types because there is no `if (BuildConfig.DEBUG)` guard. The `BASE_URL` is also hardcoded with no flavor or buildConfigField abstraction, making staging/production builds impossible without code changes.

## Wallet address disconnect in ClaimScreen

The `ClaimScreen` asks the user to enter a Solana wallet address, validates it against a base58 regex, and on success calls `viewModel.claimFirst()`. But `AirdropRepository.claimFirst()` is:

```kotlin
suspend fun claimFirst(): Result<AirdropClaimResponse> {
    val response = api.claimAirdrop()  // POST airdrop/claim — no body
    ...
}
```

The wallet address the user typed is never passed anywhere. The `ApiService` does have `updateWallet(@Body request: WalletRequest)` which could be called first, but no code in `ClaimScreen` or `AirdropViewModel` invokes it. This means the user believes their wallet is linked when it is not.

## Dependency wiring and object lifecycle

Every screen creates its own dependencies inline:

```kotlin
// LoginScreen.kt
val tokenManager = remember { TokenManager(context.applicationContext) }
val viewModel: AuthViewModel = viewModel(
    factory = AuthViewModel.Factory(AuthRepository(), tokenManager)
)
```

```kotlin
// ProfileScreen.kt
val tokenManager = remember { TokenManager(context.applicationContext) }
val profileViewModel: ProfileViewModel = viewModel(...)
val authViewModel: AuthViewModel = viewModel(
    factory = AuthViewModel.Factory(AuthRepository(), tokenManager)
)
```

This pattern creates a separate `AuthViewModel` instance per screen (Login, Register, Profile all get their own). Since `viewModel()` is scoped to the NavBackStackEntry, the Profile screen's `AuthViewModel` won't share state with the Login screen's. This is functionally fine for logout (it just calls the API and clears local storage) but means the `isLoggedIn` state observed in LoginScreen won't propagate elsewhere. The absence of a shared DI graph (Hilt, Koin, or even a manual service locator) makes the wiring fragile and hard to refactor.

## Airdrop claim: no duplicate-claim protection on client

The `ClaimScreen` allows the user to press "Klaim 1000 Token" repeatedly. `claimFirst()` re-enables the button as soon as `isLoading` returns to false. The backend presumably handles duplicate claims, but the UI does not track `hasClaimed` (available in `ProfileResponse.user.hasClaimed`) to disable the button proactively. This leads to confusing UX where the user keeps claiming and sees error messages from the server.

## Repository boilerplate and error granularity

All seven repositories follow an identical pattern:

```kotlin
return try {
    val response = api.someCall(...)
    if (response.isSuccessful) {
        response.body()?.let { Result.success(it) }
            ?: Result.failure(Exception("Empty response body"))
    } else {
        Result.failure(Exception("Failed: ${response.code()}"))
    }
} catch (e: Exception) {
    Result.failure(e)
}
```

The error messages discard the response body's error detail (the server likely returns JSON with a `message` field on 4xx responses). This means the UI shows "Claim failed: 422" instead of "You have already claimed the airdrop." Parsing `response.errorBody()` into the backend's error format would significantly improve UX.

## No test coverage

The project contains zero test files — no unit tests for ViewModels, no integration tests for repositories, no UI tests for screens. The `build.gradle.kts` declares test dependencies (JUnit, Espresso, Compose UI test) but they are unused. Given that the ViewModels contain state-machine logic (game flows with multi-step API calls), the absence of tests is a material risk for regressions.
