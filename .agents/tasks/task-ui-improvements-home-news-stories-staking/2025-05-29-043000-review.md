# UI polish across Home, News, Stories, and Staking screens

A multi-screen visual improvement pass that centers the HomeScreen quick-nav icons, removes fixed height from news images, wires up story thumbnails from a new `image_path` API field, and replaces the StakingScreen with a gradient-heavy redesign including progress bars and APR badges.

Watch for: The staking progress bar relies on a hand-rolled date parser that ignores leap years and variable month lengths, producing drift of up to ~3 days per year elapsed (**confirmed**). Story thumbnails pass `imagePath` directly to Coil without a base URL prefix, which will silently fail to load if the API returns a relative path like other similar fields in this codebase (**likely**). News images now have no height constraint at all, so a tall portrait image will dominate the scroll viewport (**confirmed**).

## High-level view

NewsScreen drops the `height(180.dp)` constraint on article images and switches from `ContentScale.Crop` to `ContentScale.FillWidth`. Without any maximum height, the card can become arbitrarily tall for portrait-oriented images.

StoriesScreen gains a `Row` layout with an 80×80 dp thumbnail pulled from the new `imagePath` field on `StoryItem`. The pattern mirrors how other screens load images, except those other screens prepend `RetrofitInstance.IMAGE_BASE_URL` to relative paths — this screen does not.

StakingScreen is rewritten significantly: a gradient hero section, custom tab indicator, redesigned type cards with gradient accent strips and pill-shaped APR badges, and user staking cards with a `LinearProgressIndicator` driven by a custom `parseSimpleDate` function. The date parsing uses a rough approximation (365 days/year, 30 days/month) that accumulates error, and there's dead code in `calculateStakingProgress` that computes unused `start`/`end` variables before calling the actual parser.

<summary>Issues (4)</summary>

1. **Broken date approximation in staking progress** — `parseSimpleDate` uses 365 days/year and 30 days/month, ignoring leap years and real month lengths. For a 90-day stake starting in January, the displayed progress will be off by several percentage points. Use `java.time.LocalDate` with `ChronoUnit.DAYS` or `SimpleDateFormat` for accurate calculation.
2. **Story thumbnail missing base URL** — `StoriesScreen` passes `item.imagePath` directly to `AsyncImage`. The blade templates show `image_path` is a relative path (e.g. `storage/ai_images/...`). Other screens in this codebase prepend `RetrofitInstance.IMAGE_BASE_URL`. Without that prefix, Coil will fail silently and show nothing.
3. **Unbounded news image height** — Removing `height(180.dp)` without adding a `heightIn(max = ...)` or `aspectRatio()` constraint means a portrait image will fill the screen width at its natural aspect ratio, potentially pushing all other content below the fold.
4. **Dead variables in calculateStakingProgress** — The `start` and `end` vals computed via `replace("-", "").toLongOrNull()` are assigned but never used. They should be removed to avoid confusion.

## Story thumbnail URL construction gap

Every other screen in this app that loads images from the backend follows one of two patterns: prepend `RetrofitInstance.IMAGE_BASE_URL` (used in `NgepetGameScreen`) or conditionally check `startsWith("http")` and prepend the domain if missing (used in `IntuitionGameScreen`):

```kotlin
// IntuitionGameScreen.kt, line 620
val imageUrl = if (item.image.startsWith("http")) item.image 
    else "https://mystical-nusa.web.id/${item.image.trimStart('/')}"
```

StoriesScreen does neither:

```kotlin
// StoriesScreen.kt
item.imagePath?.let { imageUrl ->
    AsyncImage(model = imageUrl, ...)
}
```

The backend blade template references `image_path` with a leading slash (`/${Story.image_path}`), strongly suggesting the API returns a relative path. Coil will treat a bare relative path as a file URI and fail silently — no crash, just a missing image with no error visible to the user. The fix is to apply the same URL normalization pattern used elsewhere.

## Staking progress bar accuracy

`parseSimpleDate` converts a `yyyy-MM-dd` string to epoch milliseconds using this formula:

```kotlin
val daysFromEpoch = (year - 1970) * 365L + (month - 1) * 30L + day
return daysFromEpoch * 24L * 60L * 60L * 1000L
```

The 30-day month assumption causes real divergence when February is involved: Jan 1 to Mar 1 is 59 days (non-leap) but the formula gives `(2*30+1) - (0*30+1) = 60`. For a progress bar spanning dates across February, the displayed position will be off by 1-3 percentage points depending on stake duration.

The progress bar is visual-only and doesn't drive business logic, so this is a cosmetic inaccuracy rather than a correctness bug. Android provides `java.time.LocalDate.parse()` and `ChronoUnit.DAYS.between()` on API 26+ (the app targets SDK 34), and would eliminate the approximation entirely.

Additionally, `calculateStakingProgress` computes two variables that are never read:

```kotlin
val start = startDate.take(10).replace("-", "").toLongOrNull() ?: return 0f
val end = endDate.take(10).replace("-", "").toLongOrNull() ?: return 0f
```

These are leftover from an earlier implementation attempt and should be removed.

## News image height unbounded

Removing `height(180.dp)` without any replacement constraint means a 3:4 portrait image on a 400dp-wide phone renders at 533dp tall — larger than most phone screens. A `Modifier.heightIn(max = 300.dp)` combined with `ContentScale.Crop` as a fallback, or `Modifier.aspectRatio(16f / 9f)`, would preserve the "natural width" intent while preventing extreme cases.

## MysticButton import removed from StakingScreen

The diff removes the `MysticButton` import and replaces all button instances with standard Material3 `Button` and `OutlinedButton` composables with custom colors and shapes. This diverges from the project's component library (`MysticButton` is used on other screens) and may cause visual inconsistency if `MysticButton` carries shared styling like touch ripple color or minimum size that the manual replacements don't replicate.
