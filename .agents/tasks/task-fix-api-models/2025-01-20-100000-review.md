# API model alignment for MysticNusa Android app

The change rewrites data models, ApiService return types, repository unwrapping logic, and ViewModel state mapping across the entire MysticNusa Android app to match the actual JSON shapes returned by the Laravel backend controllers. Wrapper objects that didn't exist on the wire (e.g. `response.user`, `response.types`, `response.history`) have been removed, ID/answer types corrected to match the backend's serialization, and the tarot flow restructured to populate card details from the `pickCards` response rather than the `aiReading` response.

**Watch for:** Leaderboard endpoints still use a `LeaderboardResponse` wrapper with `status` and `leaderboard` fields, but the backend returns bare JSON arrays (confirmed). `LogicalStatisticsResponse` expects `total_matches` and `average_iq`, but the backend returns `total_match` and `last_iq` (confirmed). The intuition leaderboard returns a flat array of objects with `total_played`/`total_correct`/`level` keys, which do not match `LeaderboardEntry` fields `highestScore`/`score`/`rank` (confirmed).

## High-level view

Profile, staking, and news/stories changes correctly flatten the wrapper layers and will parse successfully. Game model ID types (String UUIDs for logical questions/answers, integer match IDs for trivia/logical, UUID strings for tarot sessions and intuition matches) now match what the controllers produce.

Three leaderboard/statistics endpoints remain misaligned. All game leaderboard endpoints return bare arrays directly from Eloquent collections, but `ApiService` declares them as `Response<LeaderboardResponse>` — a wrapper class expecting `{ "status": "...", "leaderboard": [...] }`. Gson will fail to map a JSON array into an object type, so leaderboard screens will render empty. The `LogicalStatisticsResponse` has two field name mismatches that will silently null out. The tarot history endpoint returns a different nested shape than the flat `List<TarotHistoryItem>` the app expects.

## Leaderboard wrapper mismatch across all games

The `ApiService` declares:

```kotlin
@GET("trivia/leaderboard")
suspend fun getTriviaLeaderboard(): Response<LeaderboardResponse>

@GET("logical/leaderboard")
suspend fun getLogicalLeaderboard(): Response<LeaderboardResponse>

@GET("intuition/leaderboard")
suspend fun getIntuitionLeaderboard(): Response<LeaderboardResponse>
```

Where `LeaderboardResponse` is:

```kotlin
data class LeaderboardResponse(
    val status: String?,
    val leaderboard: List<LeaderboardEntry>?
)
```

But the Trivia backend returns:

```php
return response()->json(
    TriviaPlayer::select('name', 'total_played', 'highest_score', ...)->get()
);
```

This produces `[{"name":"...","highest_score":...}, ...]` — a bare array at the top level. Gson will attempt to map this array into `LeaderboardResponse` (an object), fail silently, and the `Response.body()` will be null, causing the repository to return `Result.failure(Exception("Empty response body"))`.

The fix is to either change `ApiService` to `Response<List<LeaderboardEntry>>` and adjust the repository accordingly, or modify the backend controllers to wrap the array in `{ "status": "success", "leaderboard": [...] }`.

## Logical statistics field name drift

The backend's `LogicalGameController::statistics()` returns:

```json
{
    "status": "success",
    "name": "...",
    "highest_iq": 120,
    "last_iq": 115,
    "total_match": 5
}
```

The Android model expects:

```kotlin
data class LogicalStatisticsResponse(
    val status: String?,
    @SerializedName("total_matches") val totalMatches: Int?,   // backend: "total_match"
    @SerializedName("highest_iq") val highestIq: Int?,         // matches
    @SerializedName("average_iq") val averageIq: Double?       // backend: "last_iq"
)
```

`totalMatches` will be null (key `total_matches` doesn't exist; backend sends `total_match`). `averageIq` will be null (key `average_iq` doesn't exist; backend sends `last_iq`). Only `highestIq` and `status` will parse correctly.

## Intuition leaderboard field semantics

The intuition leaderboard returns:

```json
[
    {"name": "...", "total_played": 20, "total_correct": 15, "level": 3}
]
```

But `LeaderboardEntry` maps:

```kotlin
data class LeaderboardEntry(
    val name: String?,
    @SerializedName("highest_score") val highestScore: Int?,
    @SerializedName("highest_iq") val highestIq: Int?,
    val score: Int?,
    val rank: Int?
)
```

None of `total_played`, `total_correct`, or `level` have corresponding `@SerializedName` annotations in `LeaderboardEntry`. The shared `LeaderboardEntry` model attempts to be a one-size-fits-all type, but each game's leaderboard returns different field sets. This needs per-game leaderboard entry types or a union with all possible fields annotated.

## Tarot history response shape

The backend's `history()` method:

```php
$history = TarotUserStat::with('readings')
    ->where('user_id', $user->id)
    ->orderByDesc('session_date')
    ->take(10)
    ->get();
return response()->json($history);
```

This returns an array of `TarotUserStat` objects, each containing a nested `readings` array. The Android app expects `List<TarotHistoryItem>` where each item has `id`, `session_id`, `reading`, `created_at`. The shape mismatch means Gson will deserialize the top-level array but each object won't have `session_id` or `reading` at the top level — those fields live inside the nested `readings` array.
