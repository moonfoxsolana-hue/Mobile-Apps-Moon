# Games grid, floating notifications, and profile polish

The change touches five screens/viewmodels across the Mystic Nusa Android app. It reorders and renames six games, switches the game list from a single-column card list to a 2-column `LazyVerticalGrid` with Coil-loaded images, replaces the Ular Tangga game implementation with a static "Akan Hadir" (Coming Soon) placeholder, adds a floating auto-dismiss notification to ClaimScreen (matching the existing NgepetGameScreen pattern), and redesigns ProfileScreen with a gradient avatar, icon-annotated info cards, and a stats section.

**Watch for:** Race between the auto-dismiss `LaunchedEffect` and rapid successive claims that can swallow notifications (confirmed). Hardcoded image URLs with no fallback make the GamesScreen dependent on a single remote origin with no offline or error state (confirmed). The `clearMessage()`/`clearError()` pair clears state without bumping `notificationCounter`, meaning a stale counter value can prevent re-triggering (likely).

## High-level view

The GamesScreen is now a data-driven grid — a `List<GameItem>` defines order, titles, image URLs, and routes, and a `LazyVerticalGrid(Fixed(2))` renders them. The entire screen depends on six remote images with no placeholder or error drawable configured on the `AsyncImage` calls. If the CDN is slow or unreachable, the grid renders empty squares.

The ClaimScreen notification follows the same counter-keyed `LaunchedEffect` pattern as NgepetGameScreen. The implementation has a subtle timing issue: when a user triggers multiple claims before the ViewModel emits, both results arrive in the same state update and only one message is shown. Additionally, `clearMessage()` and `clearError()` don't bump the counter, so re-displaying the same message after a clear won't retrigger the LaunchedEffect.

UlartanggaGameScreen went from a full game lobby (match listing, WebSocket integration, board state) to a minimal "coming soon" stub. The only dependency is the same CDN image used elsewhere.

The logout button in ProfileScreen navigates immediately without waiting for `authViewModel.logout()` to complete, which could orphan an in-flight server-side token invalidation if the ViewModel scope is cancelled by navigation.

## GamesScreen image loading gap

The `AsyncImage` calls specify `ContentScale.Crop` and a rounded-corner clip, but pass no `placeholder`, `error`, or `fallback` parameters:

```kotlin
AsyncImage(
    model = game.imageUrl,
    contentDescription = game.title,
    modifier = Modifier
        .fillMaxWidth()
        .aspectRatio(4f / 3f)
        .clip(RoundedCornerShape(8.dp)),
    contentScale = ContentScale.Crop
)
```

On first launch or offline, every card renders as an empty 4:3 rectangle with no user feedback. A `placeholder = ColorPainter(MysticSurface)` and `error = painterResource(R.drawable.placeholder)` would prevent blank cards. The same pattern applies to the UlartanggaGameScreen "coming soon" image.

## Notification timing and counter atomicity in ClaimScreen

The auto-dismiss LaunchedEffect:

```kotlin
LaunchedEffect(uiState.notificationCounter) {
    if (uiState.claimResponse != null || uiState.error != null) {
        kotlinx.coroutines.delay(3000)
        viewModel.clearMessage()
        viewModel.clearError()
    }
}
```

`LaunchedEffect` cancels the previous coroutine when its key changes, so a rapid second claim (which bumps `notificationCounter`) does correctly cancel the first timeout and start a new 3-second window. The edge case is when multiple results arrive in a single state emission — only one message is visible.

The deeper issue is in the clear functions:

```kotlin
fun clearMessage() {
    _uiState.value = _uiState.value.copy(claimResponse = null)
}
fun clearError() {
    _uiState.value = _uiState.value.copy(error = null)
}
```

These don't bump `notificationCounter`. After auto-dismiss fires (counter = N, response = null), the counter stays at N. The next claim bumps to N+1, which works. But the direct `.value` assignment is not atomic with respect to other coroutines updating `_uiState` concurrently — `_uiState.update { it.copy(...) }` would be safer. This matches a pre-existing pattern from NgepetViewModel, so it may be a known trade-off.

## ProfileScreen logout race

```kotlin
onClick = {
    authViewModel.logout()
    navController.navigate(Screen.Login.route) {
        popUpTo(0) { inclusive = true }
    }
}
```

Navigation pops the entire back stack immediately. If `logout()` launches a coroutine in `viewModelScope` to invalidate the token server-side, that scope is cancelled when the ViewModel is cleared during navigation. The local token is likely cleared synchronously (via `tokenManager`), so the user won't get stuck, but the server session may remain active.
