# UI fixes: Staking build, image layouts, story audio, Ngepet lobby restructure, intruder dialog flow

A mixed bag of fixes across five screens. The StakingScreen drops a deprecated `tabIndicatorOffset` API call that broke the build. NewsDetailScreen and StoryDetailScreen move to natural aspect-ratio images. StoryDetailScreen gains MediaPlayer-based audio streaming. The Ngepet lobby gets a full toolbar restructure (token display, simplified top bar, new bottom toolbar with six menus, hardcoded rules phase). The intruder guessing flow is patched so a wrong guess keeps the item-selection dialog open rather than forcing a dismissal.

Watch for: MediaPlayer lifecycle has a race between `prepareAsync` and user-initiated stop (confirmed), the `isPlaying` state goes true before audio actually starts playing. Token balance fetch silently swallows all exceptions with no user feedback (confirmed). Inventory and Shop buttons in the bottom toolbar both navigate to `AVATAR_SHOP` — one of them appears to be a placeholder with no distinct destination (confirmed).

## High-level view

StoryDetailScreen introduces a MediaPlayer for audio streaming. The async preparation creates a window where the UI shows "playing" while audio hasn't started, and tapping stop during preparation calls `stop()` on an unprepared player — caught only by a blanket `catch` that leaves the player in an indeterminate state.

The Ngepet lobby restructure moves navigation menus to a collapsible bottom bar and adds a token balance fetched from the profile API. The token balance call bypasses the repository layer and uses `RetrofitInstance.api` directly, with all exceptions silently swallowed.

The intruder guess-dialog fix leaves `guessResult` null on wrong-but-continuing guesses so the result dialog doesn't appear, and re-opens the item selection dialog. The notification auto-dismiss relies on a `notificationCounter` bump with a 3-second delay that could collide with rapid successive guesses.

## MediaPlayer lifecycle in StoryDetailScreen

`isPlaying` is set to `true` immediately after `prepareAsync()` — before the prepared callback fires:

```kotlin
mediaPlayer.prepareAsync()
mediaPlayer.setOnPreparedListener { mp -> mp.start() }
isPlaying = true  // set BEFORE audio actually starts
```

If the user taps stop during this window, `mediaPlayer.stop()` is called on a player in the "preparing" state. The Android MediaPlayer state machine throws `IllegalStateException` for `stop()` when not in Started/Paused/Prepared states. The outer `catch` sets `isPlaying = false`, so the app won't crash, but the MediaPlayer enters an error state since `reset()` in the else-branch follows the thrown `stop()` and never executes.

If `prepareAsync` fails (server unreachable, 404), neither `setOnPreparedListener` nor `setOnCompletionListener` fires, and `isPlaying` stays `true` forever. There's no `setOnErrorListener` to reset the state.

## Duplicate navigation in the bottom toolbar

"Inventory" and "Shop" both call `viewModel.goToPhase(NgepetPhase.AVATAR_SHOP)`:

```kotlin
// Inventory
IconButton(onClick = { viewModel.goToPhase(NgepetPhase.AVATAR_SHOP) })
// Shop
IconButton(onClick = { viewModel.goToPhase(NgepetPhase.AVATAR_SHOP) })
```

If these are intended to be separate features (owned avatars vs. purchase new avatars), they need distinct phases. If not, one button is redundant.

## Token balance fetch bypasses repository pattern

`loadTokenBalance()` calls `RetrofitInstance.api.getProfile()` directly instead of routing through a repository. This makes the ViewModel directly depend on the Retrofit singleton — inconsistent with the architecture used by every other network call in this class.

The empty `catch (_: Exception) { }` means network failures, auth errors, and deserialization issues all silently result in showing "0" tokens. A player checking their balance before staking tokens in a game sees an incorrect number with no error indication.

## Intruder guess-dialog keep-open behavior

When `isCorrect == false && isEnd != true`, the wrongly-guessed item is added to `guessedItemNames` and greyed out in the dialog, `showGuessItemDialog` is set back to `true`, and the floating notification shows "Tebakan salah! Coba lagi."

The `notificationCounter` bump triggers a `LaunchedEffect` that waits 3 seconds then calls `clearMessage()`. If the user guesses wrong again within 3 seconds, the counter bumps again, launching a new `LaunchedEffect`. The previous coroutine's delay fires and clears the newer message early because `clearMessage()` unconditionally nulls the message field. The UX impact is minor (notification disappears early) but observable with rapid tapping.

## Image URL construction in StoryDetailScreen

The thumbnail URL is `"https://mystical-nusa.web.id/${storyItem.imagePath}"`. If `imagePath` starts with a leading slash, this produces a double-slash (`web.id//images/...`) that could cause 404s depending on server configuration. Worth verifying the backend's stored path format.
