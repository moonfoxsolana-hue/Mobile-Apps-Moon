# Ngepet game enhancements + API route fixes + Staking type fix

Three fixes bundled in one changeset: two missing API routes that left News/Stories screens blank, a Staking model type mismatch that caused deserialization crashes, and a substantial set of Ngepet game UX enhancements (guess animation, attempt tracking, host guess re-open flow, intruder tabs).

The API route and staking fixes are straightforward. The Ngepet enhancements introduce client-side guess tracking, a pulsing animation overlay, tabbed intruder views, and a `shouldReopenGuess` mechanism for auto-reopening the host's guess dialog. The design relies on transient ViewModel state that is never reset between matches and never persisted â€” creating gaps around session continuity.

**Watch for:** Client-side guess counter drifts from server truth on app resume or match refresh (confirmed). Guess button remains enabled after all attempts are exhausted, allowing the user to fire requests the server will reject (confirmed). The `shouldReopenGuess` mechanism fires for both host and intruder roles via a shared `clearGuessResult()` path (likely).

## High-level view

The Ngepet changes add a client-side guess counter (`intruderGuessCount`) that is never reset when a match is reloaded or a new match begins. Since `loadMatchDetail` does not reset the counter, any app resume, background return, or explicit refresh will leave the counter stale â€” potentially showing "5 remaining" when the server knows the intruder has already used 3.

The guess button ("Tebak Token Host") has no `enabled` guard based on remaining attempts. The counter is displayed but never acts as a gate, so the user can keep firing guess requests past the limit until the server returns an error or `isEnd=true`.

The `shouldReopenGuess` + `lastGuessedIntruderId` pattern uses a `LaunchedEffect` keyed on `shouldReopenGuess` in `HostMatchView`. `clearGuessResult()` sets `shouldReopenGuess` when `lastGuessedIntruderId` is non-null and the game hasn't ended. Since `lastGuessedIntruderId` is only set by `hostGuess()`, the intruder path never triggers the re-open. The coupling is implicit and easy to break if future code sets `lastGuessedIntruderId` from the intruder flow.

The intruder tab filtering uses an allowlist (`status == "join" || status == "wait"` for active, `status == "end"` for history). Any future API status value would cause intruders to vanish from both tabs.

## Guess counter drift and missing reset

`intruderGuessCount` starts at 0 in `NgepetUiState` and is incremented in `intruderGuessHidden()` on each successful response. It is never reset. The relevant places where a reset should occur:

- `loadMatchDetail()` â€” called on match join, resume, and explicit refresh. Currently sets `maxGuessAttempts` based on difficulty but does not touch `intruderGuessCount`.
- `checkActiveMatch()` â€” called at ViewModel init. Loads the match detail, which again doesn't reset the counter.
- `joinMatch()` â€” starts a new intruder session but doesn't zero the counter.

```kotlin
// NgepetViewModel.kt, loadMatchDetail success handler:
_uiState.value = _uiState.value.copy(
    isLoading = false,
    matchDetail = response,
    maxGuessAttempts = maxAttempts
    // intruderGuessCount not mentioned â€” stale value persists
)
```

If the API's guess response included a "guess_count" or "remaining_attempts" field, the counter could be server-authoritative. Without that, the counter should at minimum be reset to 0 in `loadMatchDetail` (accepting that it will under-count on resume, which is better than over-counting and showing negative remaining values).

## Guess button lacks an enabled guard

In `IntruderMatchView`, the guess button is always clickable:

```kotlin
Button(
    onClick = { showGuessHiddenDialog = true },
    // no `enabled = remaining > 0` parameter
    ...
)
```

The "Sisa tebakan" text above it shows the remaining count and turns red at `remaining <= 1`, but there's no functional gate. The user can open the item selection dialog and submit a guess even when the displayed remaining is 0 or negative. The server presumably rejects this, but the UX is confusing â€” the button should be disabled or hidden when `remaining <= 0`.

## Host guess re-open mechanism

The flow:
1. Host clicks "Tebak" on an intruder â†’ `showGuessDialog = intruderId`
2. Host selects an item â†’ `viewModel.hostGuess(intruderId, itemName)` â†’ sets `lastGuessedIntruderId = intruderId`
3. Guess result comes back â†’ dialog shows result
4. User dismisses â†’ `clearGuessResult()` â†’ if `!isEnd && lastGuessedIntruderId != null` â†’ `shouldReopenGuess = true`
5. `LaunchedEffect(uiState.shouldReopenGuess)` triggers â†’ sets `showGuessDialog = lastGuessedIntruderId` â†’ calls `clearReopenGuess()`

The potential fragility: `clearGuessResult()` is called from a single shared `onDismiss` handler at the top of the composable (line 105), which fires for both host and intruder guess results. For the intruder, `lastGuessedIntruderId` is null (never set by intruder code), so `shouldReopen` evaluates to false â€” but this relies on an invariant that isn't documented or asserted.

## Intruder tab filtering

Active intruders are filtered by `status == "join" || status == "wait"`. History by `status == "end"`. If the API introduces additional statuses (e.g., "timeout", "disconnected"), those intruders would disappear from both tabs. A safer approach: define active as `status != "end"` (i.e., invert the history condition to be the allowlist).
