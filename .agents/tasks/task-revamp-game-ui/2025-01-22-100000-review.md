# Game UI revamp: backgrounds, sound, and per-game theming

Four game screens (Intuition, Logical, Trivia, Tarot) receive a visual overhaul: remote background images via Coil, per-game accent colors, sound effects and BGM through a new SoundManager singleton, decorative countdown bars, and floating animations. The approach isolates all changes to the UI/presentation layer — ViewModel calls and game logic remain untouched. Shared infrastructure (SoundManager, GameBackground, Color constants) is factored into reusable components.

**Watch for:** SoundManager is a process-global singleton with no lifecycle awareness — MediaPlayer instances for SFX can leak if the activity is destroyed mid-prepare (confirmed). The decorative countdown timers in Intuition and Trivia tick independently of actual game timing, which could confuse players if network latency makes them visually desynchronize from server-side round progression (likely). No placeholder or error handling is shown when background images fail to load (confirmed).

## High-level view

SoundManager is an `object` singleton that owns a single BGM MediaPlayer and creates one-shot MediaPlayers per sound effect. BGM lifecycle is managed per-screen via LaunchedEffect/DisposableEffect pairs. The singleton design means only one screen's BGM can play at a time, which works for the current navigation model but would silently break under parallel compositions.

GameBackground is a thin Coil AsyncImage wrapper with a vertical gradient overlay. It provides no placeholder or error state — on slow networks or 404s the user sees only the dark overlay with no indication a background was expected.

The countdown bars in Intuition and Trivia are purely decorative coroutine-driven timers that don't interact with game logic, but they tick on a fixed 10s clock unrelated to server round timing, creating potential visual confusion under latency.

The Tarot screen replaces emoji card placeholders with AsyncImage loading `closed-card.png` and shows revealed card images in ORACLE/READING phases, using a `/`-prefix URL normalization pattern that differs slightly from IntuitionGameScreen's `trimStart('/')` approach — an inconsistency that could bite if relative paths arrive in unexpected formats.

<details>
<summary>Issues (5)</summary>

1. **SFX MediaPlayer leak on activity destruction** — If the Activity/Process is killed while `prepareAsync()` is in flight, the SFX MediaPlayer is never released. Hold a weak reference or tie SFX players to a lifecycle-aware scope, or maintain a list and release on `stopBgm()` / a dedicated `releaseAll()`.
2. **No image load-failure UX** — GameBackground and all AsyncImage usages lack `placeholder` and `error` drawables. On slow networks or 404s, users see empty space with no affordance. Add a dark fallback or shimmer placeholder.
3. **BGM singleton is not multi-screen safe** — If two screens are ever composed simultaneously (e.g., shared-element transition, dialog overlay), the second `playBgm` kills the first without the first screen's DisposableEffect running. Document this constraint or guard against it.
4. **Countdown bar drift vs. server time** — The 10-second decorative countdown in Intuition/Trivia is client-only and unrelated to when the server actually advances the round. Under latency the bar can hit zero while the round is still active, or a new round can start before the bar finishes. Consider resetting the bar on state changes rather than a fixed 10s timer.
5. **IntuitionGameScreen sound triggers on initial composition** — `LaunchedEffect(uiState.lastAnswerCorrect)` fires on first composition when `lastAnswerCorrect` is `null`, hitting the `else` no-op branch. If the ViewModel ever initializes with a non-null value (e.g., state restoration), a spurious sound would play. Guard with a `snapshotFlow` that skips the initial emission.

## SoundManager lifecycle and resource leaks

SoundManager creates a new `MediaPlayer` per SFX call and relies on `onCompletionListener` to release it, but `prepareAsync()` is non-blocking — if the composable leaves composition (activity destruction, config change) before the player reaches `onCompletion`, the player leaks. There's no collection of active SFX players and no way to release them externally.

```kotlin
// SoundManager.kt line 22-36
fun playSound(context: Context, url: String) {
    if (isMuted) return
    val player = MediaPlayer().apply {
        setDataSource(url)
        setOnPreparedListener { it.start() }
        setOnCompletionListener { it.release() }
        setOnErrorListener { mp, _, _ -> mp.release(); true }
        prepareAsync()
    }
    // `player` is now unreferenced — only the internal callbacks hold it
}
```

The `player` local variable is immediately discarded. Within a single-process lifecycle (config change during network-loading a large audio file), this is a real leak vector. A `mutableListOf<MediaPlayer>()` tracking active SFX players, cleaned up via a `releaseAll()` method called from screen disposal, would close the gap.

## Background image loading without fallback

`GameBackground` uses `AsyncImage` with no `placeholder`, `error`, or `fallback` parameters:

```kotlin
AsyncImage(
    model = imageUrl,
    contentDescription = null,
    contentScale = ContentScale.Crop,
    modifier = Modifier.fillMaxSize()
)
```

On first load (cold cache), or on a 404/timeout, the background area is transparent — the dark gradient overlay still renders, so it won't be a white flash, but "loaded" and "failed" are visually indistinguishable from "still loading." Users on slow connections see a dark screen with no indication that a background should appear. The same gap applies to card-back images in TarotGameScreen and item images in IntuitionGameScreen.

## Decorative countdown bar drift

Both IntuitionGameScreen and TriviaGameScreen implement a 10-second countdown bar using a coroutine that decrements every 50ms:

```kotlin
LaunchedEffect(uiState.currentRound) {
    countdownProgress = 1f
    val totalDuration = 10_000L
    val stepDelay = 50L
    val steps = totalDuration / stepDelay
    val decrement = 1f / steps
    repeat(steps.toInt()) {
        delay(stepDelay)
        countdownProgress = (countdownProgress - decrement).coerceAtLeast(0f)
    }
}
```

The Intuition version resets on `uiState.currentRound` changes; the Trivia version resets on `uiState.currentQuestion` changes. If the network round-trip for submitting an answer takes 2-3 seconds, the bar continues counting during that period and may visually reach zero before the next round appears. Players could interpret this as a timeout when it's actually just waiting for the server response.

## Tarot card image URL normalization inconsistency

TarotGameScreen handles card images that might arrive with a leading `/`:

```kotlin
card.image?.let { imageUrl ->
    val fullUrl = if (imageUrl.startsWith("/")) {
        "https://mystical-nusa.web.id$imageUrl"
    } else {
        imageUrl
    }
    AsyncImage(model = fullUrl, ...)
}
```

IntuitionGameScreen uses a different normalization: `"https://mystical-nusa.web.id/${item.image.trimStart('/')}"`. If a card image arrives as a relative path without a leading slash (e.g., `images/tarot/foo.png`), the Tarot version treats it as an absolute URL and fails, while Intuition's approach would correctly prepend the base. Worth standardizing on one pattern across all screens.

## Sound trigger timing in IntuitionGameScreen

```kotlin
LaunchedEffect(uiState.lastAnswerCorrect) {
    when (uiState.lastAnswerCorrect) {
        true -> SoundManager.playSound(context, ".../correct-intuition.wav")
        false -> SoundManager.playSound(context, ".../wrong.mp3")
        else -> { /* no-op */ }
    }
}
```

This fires whenever `lastAnswerCorrect` changes value, including on initial composition (where it's `null`, hitting the no-op branch). The Trivia screen gates its sound on `uiState.showFeedback` being true, which is a tighter condition. If the Intuition ViewModel ever stores `lastAnswerCorrect` in SavedStateHandle and restores it on process recreation, the sound would fire on recomposition — a spurious audio cue with no user action.
